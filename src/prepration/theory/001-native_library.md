To give you a clear, side-by-side comparison, let’s build a simple native module called **`DeviceCrypto`** that exposes a single synchronous method: `sha256(message: string)`.

This is a perfect example for an IBM interview because cryptographic hashing is a heavy operation that benefits from native speed, and it highlights exactly how the **Old Architecture (Bridge)** differs from the **New Architecture (JSI / TurboModules)**.

---

## 1. The Legacy Way: The Old Architecture (The Bridge)

In the old architecture, data must be serialized into JSON, passed asynchronously across the bridge, deserialized on the native side, executed, and sent back the same way.

### Android Implementation (Java)

First, you create the Module class:

```java
package com.devicecrypto;

import com.facebook.react.bridge.ReactApplicationContext;
import com.facebook.react.bridge.ReactContextBaseJavaModule;
import com.facebook.react.bridge.ReactMethod;
import com.facebook.react.bridge.Promise;
import java.security.MessageDigest;

public class DeviceCryptoModule extends ReactContextBaseJavaModule {

    public DeviceCryptoModule(ReactApplicationContext reactContext) {
        super(reactContext);
    }

    // This defines the name used in JavaScript: NativeModules.DeviceCrypto
    @Override
    public String getName() {
        return "DeviceCrypto";
    }

    // Exposing the method asynchronously via the Bridge using Promises
    @ReactMethod
    public void sha256(String message, Promise promise) {
        try {
            MessageDigest digest = MessageDigest.getInstance("SHA-256");
            byte[] hash = digest.digest(message.getBytes("UTF-8"));
            StringBuilder hexString = new StringBuilder();
            for (byte b : hash) {
                String hex = Integer.toHexString(0xff & b);
                if(hex.length() == 1) hexString.append('0');
                hexString.append(hex);
            }
            promise.resolve(hexString.toString());
        } catch (Exception e) {
            promise.reject("SHA256_ERROR", e.getMessage());
        }
    }
}

```

Then, you must wrap it in a `ReactPackage` and register it in your `MainApplication.java`.

### JavaScript Usage (Old)

```javascript
import { NativeModules } from "react-native";

const { DeviceCrypto } = NativeModules;

// Calling it requires an asynchronous await because it crosses the Bridge
async function hashData() {
  try {
    const hashedString = await DeviceCrypto.sha256("Hello IBM");
    console.log(hashedString);
  } catch (error) {
    console.error(error);
  }
}
```

## 2. The Modern Way: The New Architecture (TurboModules & JSI)

In the New Architecture, we eliminate the asynchronous JSON bridge entirely. JavaScript invokes the native C++ method **synchronously and directly** via JSI memory references.

### Step A: Define the Type Specification (TypeScript)

You must create a strictly typed specification file. The file name must start with `Spec` (e.g., `SpecDeviceCrypto.ts`).

```typescript
import { TurboModule, TurboModuleRegistry } from "react-native";

export interface Spec extends TurboModule {
  // Note: This can now return a synchronous string directly! No Promise required!
  sha256(message: string): string;
}

export default TurboModuleRegistry.getEnrolled<Spec>(
  "DeviceCrypto",
) as Spec | null;
```

### Step B: The Codegen Layer (Automatic)

When you build the app, React Native's **Codegen** scans this file and automatically generates C++ code and an abstract Java class named `NativeDeviceCryptoSpec`.

### Step C: Android Implementation (Java/Kotlin)

Instead of extending `ReactContextBaseJavaModule`, your native class extends the **Codegen-generated specification class**.

```java
package com.devicecrypto;

import androidx.annotation.NonNull;
import com.devicecrypto.NativeDeviceCryptoSpec; // Generated by Codegen
import com.facebook.react.bridge.ReactApplicationContext;
import java.security.MessageDigest;

public class DeviceCryptoModule extends NativeDeviceCryptoSpec {

    public static final String NAME = "DeviceCrypto";

    public DeviceCryptoModule(ReactApplicationContext reactContext) {
        super(reactContext);
    }

    @Override
    @NonNull
    public String getName() {
        return NAME;
    }

    // Implements the synchronous signature defined in TypeScript
    @Override
    public String sha256(String message) {
        try {
            MessageDigest digest = MessageDigest.getInstance("SHA-256");
            byte[] hash = digest.digest(message.getBytes("UTF-8"));
            StringBuilder hexString = new StringBuilder();
            for (byte b : hash) {
                String hex = Integer.toHexString(0xff & b);
                if(hex.length() == 1) hexString.append('0');
                hexString.append(hex);
            }
            return hexString.toString(); // Returns directly to JS instantly
        } catch (Exception e) {
            return "";
        }
    }
}

```

### JavaScript Usage (New)

```javascript
// Import directly from your typed specification file
import DeviceCryptoSpec from "./SpecDeviceCrypto";

function hashData() {
  // Synchronous execution! It behaves exactly like a native JS function
  // because JS has a direct memory pointer to the C++ translation layer.
  const hashedString = DeviceCryptoSpec?.sha256("Hello IBM");
  console.log(hashedString);
}
```

---

## 💡 Key Takeaways for your IBM Interview:

1. **Type Safety:** The Old Architecture relies on arbitrary string lookups (`NativeModules.DeviceCrypto`) which can fail at runtime if misspelled. The New Architecture enforces compile-time type safety via **Codegen**.
2. **Synchronous Execution:** In the Old Architecture, you had to use Promises or Callbacks because the Bridge is completely asynchronous. TurboModules allow **direct synchronous execution** for performance-critical functions.
3. **Memory/Performance:** The Old Architecture copies and serializes data into strings over a queue. The New Architecture uses C++ host objects, making data transfers significantly cheaper and faster.
